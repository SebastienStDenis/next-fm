# Phase 2: authentication and signup (Supabase Auth)

*Written 2026-07-08 by Claude (Fable 5).*

The design and implementation spec for Phase 2 of
`docs/2026-07-08-production-deployment-plan.md`: real signup and login via
Supabase Auth, an API that only authenticated users can call, and the local
stack moving from the compose Postgres to the Supabase CLI stack so local
development mirrors production auth exactly.

This doc is written to be executed step by step. Code shown in full is meant to
be used as-is; everything else is a mechanical instruction (rename, move,
delete) with the exact before/after named. Work the stages in order - each
stage leaves the repo working.

**Guiding simplifications** (deliberate, revisit later):

- **Email + password only.** No Google/social login yet (the production plan
  sketched "email + Google"; Google is a follow-up, not v1).
- **No email confirmation.** Signup logs the user straight in. Turning
  confirmations on later needs an SMTP provider and a confirm route; see
  "Follow-ups."
- **No password reset flow yet.** Same reason. Follow-up.

## What this phase delivers

- Users sign up and log in with email + password. Sessions live in cookies via
  `@supabase/ssr`; the browser never handles raw tokens.
- Every backend endpoint except `/health` requires a valid Supabase-issued JWT.
  "Which user" comes from the token, never from the URL - the old
  trust-any-UUID-in-the-path scheme is gone, along with the page that listed
  and created users.
- One Supabase auth identity maps to one `User` row (`users.supabase_user_id`),
  created automatically on first authenticated request.
- Locally, `supabase start` replaces the compose Postgres for app data: same
  GoTrue auth engine, same key/URL shapes as hosted. Temporal switches to the
  single-container dev server (SQLite persistence, built-in UI), so the compose
  `db` Postgres disappears entirely.

## Design

### How the API gets locked down

The Render API URL is public HTTP and stays that way - what protects it is that
**every endpoint (except `/health`) demands a `Bearer` JWT signed by our
Supabase project**, and the only way to get one is to sign in through Supabase
Auth. Concretely:

- A FastAPI dependency verifies the JWT's signature, expiry, audience
  (`authenticated`) and issuer on every request, then resolves it to our `User`
  row. No token, expired token, or token signed by anyone else → `401`.
- **Authorization is structural.** Endpoints move from `/users/{user_id}/...`
  to `/me/...`: the acting user is derived from the verified token, so there is
  no user id input to tamper with and no cross-user access to check for.
- The browser never calls the API directly. All calls run server-side in Next
  (server components, server actions, route handlers) and attach the session's
  access token. `API_URL` stays server-only on Vercel; `CORS_ORIGINS` stays as
  a belt-and-braces backstop.
- Signup itself is the remaining public surface, and it is Supabase's, not
  ours: GoTrue applies its own rate limits (per-IP signup, token endpoints).
  If the site should ever be invite-only, flip **Allow new users to sign up**
  off in the Supabase dashboard - no code change.

Things deliberately *not* added: a shared secret header between Vercel and
Render (redundant once every request carries a verified JWT), Render private
networking (Vercel isn't in it), and per-user rate limiting on the expensive
sync endpoints (follow-up; auth already stops anonymous abuse).

### Identity mapping

`users` gains a nullable, unique `supabase_user_id` column holding the Supabase
auth user id (the JWT's `sub` claim).

- **Just-in-time provisioning**: the auth dependency looks up
  `User.supabase_user_id == sub`; if no row exists it creates one, taking the
  name from the JWT's `user_metadata.display_name` (set at signup), falling
  back to the email local-part. No separate "complete your profile" step.
- The column stays **nullable**: pre-auth rows (the local demo user, the
  existing production row) simply become unreachable until linked or deleted.
  Linking the existing production user is a one-time SQL step in the cutover
  runbook (Stage D).
- `LastfmConnection` and everything downstream keys off `users.id` and is
  untouched. The Spotify bot account is untouched - auth is "who is logged
  into our site," not "connect your Spotify."

### Token verification: HS256 and JWKS, dispatched by header

Supabase projects sign access tokens one of two ways, and we will encounter
both:

- **Local CLI stack**: HS256 with the project JWT secret (the CLI's well-known
  default unless overridden).
- **Hosted projects on JWT signing keys** (the current default for new
  projects): asymmetric ES256/RS256, public keys published at
  `{SUPABASE_URL}/auth/v1/.well-known/jwks.json`.

The backend inspects the (unverified) token header's `alg` and dispatches:
`HS256` → verify with `SUPABASE_JWT_SECRET` (rejected outright if that setting
is empty, so a well-known local secret can never be accepted in production);
`ES256`/`RS256` → verify against the cached JWKS via PyJWT's `PyJWKClient`.
Anything else → `401`. One code path, both environments, no per-environment
flags.

One wrinkle from containers: inside compose, the backend reaches local Supabase
at `host.docker.internal:54321`, but locally-issued tokens carry
`iss = http://127.0.0.1:54321/auth/v1`. So the issuer to validate is its own
setting (`SUPABASE_ISSUER`), defaulting to `{SUPABASE_URL}/auth/v1` and
overridden only by compose.

### Request flow

```
browser ──form POST──▶ Next server action ──signInWithPassword──▶ Supabase Auth
   ◀── session cookies set (@supabase/ssr) ──┘

browser ──GET /dashboard──▶ Next proxy (refresh session; redirect if signed out)
                              └▶ server component ──apiFetch()──▶ FastAPI
                                   Authorization: Bearer <access_token>
                                        └▶ verify JWT ──▶ resolve/create User ──▶ data
```

### B2. `backend/app/auth.py` (new file, complete)

```python
import uuid
from functools import lru_cache
from typing import Annotated

import jwt
from fastapi import Depends, HTTPException
from fastapi.security import HTTPAuthorizationCredentials, HTTPBearer
from pydantic import BaseModel
from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession

from app.config import Settings, get_settings
from app.db import get_session
from app.models import User

ALLOWED_ALGORITHMS = ("HS256", "ES256", "RS256")


class Claims(BaseModel):
    sub: uuid.UUID
    email: str | None = None
    display_name: str | None = None


@lru_cache
def _jwks_client(jwks_url: str) -> jwt.PyJWKClient:
    return jwt.PyJWKClient(jwks_url)


def _unauthorized(detail: str) -> HTTPException:
    return HTTPException(status_code=401, detail=detail, headers={"WWW-Authenticate": "Bearer"})


def verify_token(token: str, settings: Settings) -> Claims:
    """Verify a Supabase-issued JWT and distill the claims the app uses.

    Dispatches on the token's declared algorithm: HS256 verifies against the
    shared JWT secret (rejected when none is configured), asymmetric
    algorithms verify against the project's published JWKS.
    """
    issuer = settings.supabase_issuer or f"{settings.supabase_url}/auth/v1"
    try:
        alg = jwt.get_unverified_header(token).get("alg")
        if alg not in ALLOWED_ALGORITHMS:
            raise _unauthorized("Unsupported token algorithm")
        if alg == "HS256":
            if not settings.supabase_jwt_secret:
                raise _unauthorized("Unsupported token algorithm")
            key = settings.supabase_jwt_secret
        else:
            jwks_url = f"{settings.supabase_url}/auth/v1/.well-known/jwks.json"
            key = _jwks_client(jwks_url).get_signing_key_from_jwt(token).key
        payload = jwt.decode(
            token, key, algorithms=[alg], audience="authenticated", issuer=issuer
        )
    except jwt.PyJWTError:
        raise _unauthorized("Invalid or expired token") from None
    sub = payload.get("sub")
    if sub is None:
        raise _unauthorized("Invalid or expired token")
    metadata = payload.get("user_metadata") or {}
    return Claims(sub=sub, email=payload.get("email"), display_name=metadata.get("display_name"))


_bearer = HTTPBearer(auto_error=False)


def get_claims(
    credentials: Annotated[HTTPAuthorizationCredentials | None, Depends(_bearer)],
) -> Claims:
    # Sync on purpose: FastAPI runs it in the threadpool, so the blocking
    # JWKS fetch inside PyJWKClient never stalls the event loop.
    if credentials is None:
        raise _unauthorized("Not authenticated")
    return verify_token(credentials.credentials, get_settings())


async def get_current_user(
    claims: Annotated[Claims, Depends(get_claims)],
    session: Annotated[AsyncSession, Depends(get_session)],
) -> User:
    """Resolve the token to our User row, creating it on first login."""
    result = await session.execute(select(User).where(User.supabase_user_id == claims.sub))
    user = result.scalar_one_or_none()
    if user is None:
        name = claims.display_name or (claims.email or "").split("@")[0] or "Music lover"
        user = User(supabase_user_id=claims.sub, name=name)
        session.add(user)
        await session.commit()
    return user


CurrentUserDep = Annotated[User, Depends(get_current_user)]
```

### B3. `backend/app/supabase_admin.py` (new file, complete)

Deleting an account must also delete the auth user, or the next login would
silently re-provision an empty account. GoTrue's admin API does it; the secret
key authorizes it.

```python
import uuid

import httpx


class SupabaseAdminError(Exception):
    pass


class SupabaseAdminClient:
    """Minimal GoTrue admin client, authorized by the project secret key."""

    def __init__(self, url: str, secret_key: str) -> None:
        self._client = httpx.AsyncClient(
            base_url=f"{url}/auth/v1",
            headers={"apikey": secret_key, "Authorization": f"Bearer {secret_key}"},
        )

    async def delete_user(self, supabase_user_id: uuid.UUID) -> None:
        try:
            response = await self._client.delete(f"/admin/users/{supabase_user_id}")
        except httpx.HTTPError as exc:
            raise SupabaseAdminError(f"Supabase admin API unreachable: {exc}") from exc
        if response.status_code == 404:
            return
        if response.is_error:
            raise SupabaseAdminError(
                f"Failed to delete auth user: {response.status_code} {response.text}"
            )

    async def aclose(self) -> None:
        await self._client.aclose()
```

import { createServerClient } from "@supabase/ssr";
import { cookies } from "next/headers";

import { supabaseKey, supabaseUrl } from "./env";

export async function createClient() {
  const cookieStore = await cookies();

  return createServerClient(supabaseUrl, supabaseKey, {
    cookies: {
      getAll() {
        return cookieStore.getAll();
      },
      setAll(cookiesToSet) {
        try {
          cookiesToSet.forEach(({ name, value, options }) =>
            cookieStore.set(name, value, options),
          );
        } catch {
          // Called from a server component; the proxy refreshes sessions.
        }
      },
    },
  });
}

### C2. Session refresh + route protection: `frontend/src/proxy.ts`

```ts
import { createServerClient } from "@supabase/ssr";
import { NextResponse, type NextRequest } from "next/server";

import { supabaseKey, supabaseUrl } from "@/lib/supabase/env";

const PUBLIC_PATHS = new Set(["/", "/login", "/signup", "/about"]);

export async function proxy(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request });

  const supabase = createServerClient(supabaseUrl, supabaseKey, {
    cookies: {
      getAll() {
        return request.cookies.getAll();
      },
      setAll(cookiesToSet) {
        cookiesToSet.forEach(({ name, value }) =>
          request.cookies.set(name, value),
        );
        supabaseResponse = NextResponse.next({ request });
        cookiesToSet.forEach(({ name, value, options }) =>
          supabaseResponse.cookies.set(name, value, options),
        );
      },
    },
  });

  // Refreshes the session if expired; nothing may run between client
  // creation and this call, per @supabase/ssr docs.
  const {
    data: { user },
  } = await supabase.auth.getUser();

  const { pathname } = request.nextUrl;
  // API routes answer JSON; they handle missing sessions themselves rather
  // than redirecting a fetch() to an HTML login page.
  if (!user && !PUBLIC_PATHS.has(pathname) && !pathname.startsWith("/api")) {
    const url = request.nextUrl.clone();
    url.pathname = "/login";
    return NextResponse.redirect(url);
  }
  if (user && (pathname === "/login" || pathname === "/signup")) {
    const url = request.nextUrl.clone();
    url.pathname = "/dashboard";
    return NextResponse.redirect(url);
  }

  // Must return this exact response object so refreshed cookies survive.
  return supabaseResponse;
}

export const config = {
  matcher: ["/((?!_next/static|_next/image|favicon.ico|icon.svg|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)"],
};
```

### C3. Authenticated API helper: `frontend/src/lib/api.ts`

Every backend call goes through this; it replaces the scattered
`process.env.API_URL ?? "http://localhost:8000"` constants.

```ts
import "server-only";

import { redirect } from "next/navigation";

import { createClient } from "@/lib/supabase/server";

const apiUrl = process.env.API_URL ?? "http://localhost:8000";

export async function apiFetch(
  path: string,
  init: RequestInit = {},
): Promise<Response> {
  const supabase = await createClient();
  const {
    data: { session },
  } = await supabase.auth.getSession();
  if (!session) {
    // The proxy guards pages, so this only fires on expiry races.
    redirect("/login");
  }
  return fetch(`${apiUrl}${path}`, {
    ...init,
    headers: {
      ...init.headers,
      Authorization: `Bearer ${session.access_token}`,
    },
  });
}
```

(`getSession` reads the cookie without a network round trip; the backend is
the verifier. The proxy already refreshed it this request.)
